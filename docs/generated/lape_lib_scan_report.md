## Known Limitations

1. **Declaration deduplication**: The Lape source files use a documentation comment pattern where function signatures appear twice — once in a doc-comment block (prefixed with `(*`) and once as the actual declaration. The extractor deduplicates by normalizing the signature string, but some near-duplicate entries may remain where parameter formatting differs between the doc and actual declaration.

2. **Multi-line type blocks**: Type blocks (`type ... end`) that span many lines may not yield all nested type aliases, since the extractor relies on block-context tracking (entering/exiting `const`, `var`, `type` blocks). Deeply nested or unusual formatting may cause some type aliases to be missed.

3. **Constant extraction**: Only constants with names matching an all-caps or CamelCase pattern were captured, to avoid capturing local expressions inside const blocks. Constants with all-lowercase names may be missed.

4. **Global variable detection**: Module-level `var` blocks are detected by the `var` keyword alone on a line. Variable declarations inside `begin ... end` blocks or inline with code are not captured. **Critically: global singletons wrapped in `{$H-}name: Type;{$H+}` compiler-hint directives were missed entirely** — `Map: TRSMap` (map.simba:1532) and `Banks: TRSBanks` (mapobjectarray.simba:708) were both missed and have been manually added. Other `{$H-}...{$H+}` wrapped variables not yet verified may also be missing.

5. **Record field summary quality**: The 1250 record fields extracted in the third pass have mechanically-generated summaries (`Field of RecordName. Type: T.`). These are accurate but not descriptive — they do not explain the purpose or valid values of each field. The 11 manually added fields from the `TRSMapObject`/`TRSObjectV2`/`TRSNPCV2` family have richer summaries written by hand.

5. **Summary quality**: Summaries are derived mechanically from return type or keyword patterns. Functions whose purpose is only clear from the body (which is not read) are marked `unclear from source` in the uncertainty field when appropriate, or given a generic summary based on the function name.

7. **`SetupEx` missed by initial scan**: Grep patterns targeted `function`/`procedure` declarations beginning with a type method (`T*.Method`). `SetupEx` exists as a `procedure` form (not returning a value) and its name was not in the initial pattern set. The three `SetupEx` procedures for `TRSObjectV2` and `TRSNPCV2` have been manually added. Other `*Ex` procedure variants may also be missing.

8. **include directives**: Includes are captured only when in `{$I filename}` format. Other compiler directive variants or dynamic includes are not captured.

7. **Null lines in WaspLib**: A small number of null/empty lines in some WaspLib files triggered non-fatal PowerShell warnings during extraction; these lines were skipped safely with a null guard.
